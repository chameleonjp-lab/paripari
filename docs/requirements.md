## 1. プロダクト概要（GD）

### 1.1 一言で
敵の攻撃に合わせて、**攻撃が来た方向と反対側のボタンを「ちょうど」のタイミングで1回タップ**し、
ギリギリで受け流す（パリィする）ゲーム。

### 1.2 体験のコア
- 「今だ！」と感じた一瞬に正しい方向へ弾く**緊張感**と、決まったときの**爽快感**。
- 1回の判定が短く、成功すれば気持ちよく、失敗しても即やり直せる**高速リトライ性**。
- 操作はほぼタップのみ。初見でも5秒で理解できる導線。

### 1.3 名称の由来
「パリィ（parry）」＋受け流しの擬音「パリン」。和の居合（侍）をモチーフにした
クリスプな打撃感を表現する。

### 1.4 ターゲット
- カジュアル〜ミッドコアのスマホブラウザユーザー。
- 1プレイ 30秒〜2分の短時間反復プレイ。スキマ時間・ワンハンド（片手親指）操作。

### 1.5 プラットフォーム / 動作対象 ✅
- **必須**: iPhone SE（第2/第3世代, 375×667 CSS px）の Safari（iOS 15+）。
- **必須**: iPhone SE（第1世代, 320×568）でも崩れず遊べること（最小対応解像度）。
- 推奨対応: iOS Chrome、Android Chrome、PC ブラウザ（キーボード操作可）。
- インストール不要。URL を開けば即プレイ（PWA 化は ⏳ v2）。

## 2. ゲームメカニクス（GD）

### 2.1 用語定義
| 用語 | 定義 |
| --- | --- |
| 敵（Enemy） | 攻撃を仕掛けてくる対象。画面上方/左右から出現。 |
| 攻撃（Attack） | 一方向から「侍（プレイヤー）」へ向かう斬撃。 |
| 予兆（Telegraph） | 攻撃方向を示す前兆演出（光・矢印・色）。 |
| パリィライン | 攻撃が到達し受け流し可能になる基準線（プレイヤー位置）。 |
| インパクト時刻 T | 攻撃がパリィラインに到達する瞬間。判定窓の中心。 |
| 判定窓 | T を中心とした成功許容時間帯。 |
| コンボ | 連続成功数。途切れると 0 に戻る。 |

### 2.2 コアループ ✅
```
出現 → 予兆(方向提示) → 接近(攻撃が線へ向かう) → [インパクトT] → 判定 → 演出 → 次へ
```
1. 敵が攻撃方向を**予兆**で提示（例: 左から斬撃）。
2. プレイヤーは到達直前に「反対側」ボタンをタップ（左攻撃→右ボタン）。
3. タップ時刻と T の差・方向の正誤で **3段階判定**。
4. 結果演出（弾き火花/被弾）→ 次の攻撃へ。

### 2.3 入力と方向の対応 ✅
- v1 のコア難易度は **左右2方向**。攻撃が来た**反対側**のボタンを押す。
  - 左から攻撃 → **右ボタン**で受け流す。
  - 右から攻撃 → **左ボタン**で受け流す。
- 「反対側を押す」反転がコアの緊張感であり、意図的に残す（GD合意）。
- 高難度で **上下を追加（4方向）** する 🔁（§5 難易度参照）。
- 1攻撃につき有効入力は**最初の1タップのみ**。連打しても2回目以降は判定済として無視。

### 2.4 判定モデル（時間） ✅
タップ時刻 `t`、インパクト時刻 `T`、誤差 `Δ = t − T`（負=早い, 正=遅い）。

| 判定 | 条件 |
| --- | --- |
| **完全成功 (PERFECT)** | 方向が正しく、`|Δ| ≤ PERFECT_WINDOW` |
| **成功 (GOOD)** | 方向が正しく、`PERFECT_WINDOW < |Δ| ≤ GOOD_WINDOW` |
| **失敗 (MISS)** | 上記以外（方向違い / 早すぎ / 遅すぎ / `T + GOOD_WINDOW` まで無入力） |

既定値（モバイル入力遅延を見込んだ余裕設定, 🔁 調整可）:
- `PERFECT_WINDOW = 60ms`（中心 ±60ms = 幅120ms）
- `GOOD_WINDOW = 140ms`（中心 ±140ms = 幅280ms）
- 早押し許容開始: `T − GOOD_WINDOW` より前の入力は「早すぎ＝MISS」だが、
  予兆前（攻撃方向確定前）の入力は判定対象外（無効＝ペナルティなし）とする。
- **方向違いは即 MISS**（タイミングが合っていても不可）。

> QA注: 入力遅延は端末・ブラウザで変動するため、実測キャリブレーション（§9.4）で
> 体感を担保。判定が厳しすぎて理不尽にならないこと（プロダクトの明示要件）。

### 2.5 ライフ / 終了条件 ✅
- プレイヤーHP = **ハート3**。MISS で 1 減少。0 で **ゲームオーバー**。
- PERFECT 連続でごく稀に回復（10 PERFECT 連続ごとにハート+1、上限3）🔁。
- ゲームオーバー → リザルト表示 → ワンタップで即リトライ。

### 2.6 反則・例外処理 ✅
- 予兆が出ていない待機中のタップ: 無効（ペナルティなし）。空打ち音のみ。
- 同一攻撃への二重入力: 2回目以降無視。
- ポーズ中/カウントダウン中の入力: ゲーム判定に影響しない。

## 3. スコアリング（GD）

### 3.1 スコア構成 ✅
1攻撃あたりの獲得点 = `基礎点 × コンボ倍率 + タイミングボーナス`

| 要素 | 値（🔁 調整可） |
| --- | --- |
| 基礎点 GOOD | 100 |
| 基礎点 PERFECT | 300 |
| コンボ倍率 | `1 + floor(combo / 5) × 0.5`（5連ごとに+0.5、上限 ×4.0） |
| タイミングボーナス | `round((1 − |Δ| / GOOD_WINDOW) × 50)`（中心ほど高い, 0〜50） |
| MISS | 0点・コンボ0リセット |

### 3.2 評価指標 ✅
- **総スコア**（最終結果のメイン指標）。
- **最大コンボ**。
- **PERFECT率**（PERFECT / 成功総数）。
- **到達ウェーブ/難易度**。
- ベストスコアは `localStorage` に保存し、リザルトで自己ベスト比較を表示 🔁。

### 3.3 ランク表示 🔁
最終スコアに応じ S / A / B / C を付与（しきい値は実装後に実測で調整）。

## 4. 画面・UI/UX 設計（UX）

### 4.1 画面一覧と遷移 ✅
```
TITLE ──tap PLAY──▶ READY(3-2-1) ──▶ PLAYING ──HP0──▶ RESULT ──tap RETRY──▶ READY
  ▲                                     │                         │
  └──────────── tap HOME ◀──────────────┴── pause ◀── tap ⏸ ──────┘
```
- **TITLE**: ロゴ「パリパリ」、PLAY、遊び方（チュートリアル）、設定、ベストスコア。
- **READY**: 3→2→1 カウントダウン。操作説明の最小ヒントを重畳。
- **PLAYING**: ゲーム本編（§4.3）。
- **PAUSE**: 半透明オーバーレイ。再開 / リトライ / ホーム。
- **RESULT**: スコア、最大コンボ、PERFECT率、ランク、ベスト更新、リトライ/シェア。

### 4.2 レイアウト原則（iPhone SE 最適化）✅
- 縦持ち固定の前提でデザイン（横持ちは「縦にしてね」案内を表示）🔁。
- 画面を縦に3分割:
  - 上部（〜45%）: 敵・攻撃の演出エリア。
  - 中央: **パリィライン**とプレイヤー（侍）。
  - 下部（〜35%, `min 220px`）: **左右2つの巨大タップボタン**。
- タップボタンは画面下半分を左右に二分。**最小タップ領域 各幅≥160px・高さ≥120px**、
  親指が自然に届く位置（ボトム寄せ）。`env(safe-area-inset-bottom)` を加味。
- 重要情報（HP/スコア/コンボ）は上部 HUD に集約し、攻撃演出と干渉させない。
- フォントは可変（vw/clamp）で 320px 幅でも破綻しない。

### 4.3 プレイ画面 HUD ✅
- 左上: ハート ♥♥♥（残HP）。
- 中央上: スコア（大）+ コンボ表示（"12 COMBO" バウンス）。
- 右上: ⏸ ポーズ。
- 判定テキスト: 中央にポップ（PERFECT=金/虹、GOOD=白/水色、MISS=赤）。
- 攻撃方向の予兆: 攻撃側エッジの発光 + 方向アイコン（矢印）。

### 4.4 ビジュアルコンセプト（UX）✅
- テーマ: **和 × ミニマル × ネオン墨**。黒〜藍のグラデ背景に、白刃・金・朱の差し色。
- 受け流し成功で**白〜金の閃光**と**火花パーティクル**、軽いスローモー（PERFECT時）。
- 被弾は**朱のフラッシュ**＋画面シェイク（弱）。
- 動きはイージング重視（`cubic-bezier`）。"ジューシー"なヒットストップを多用。

### 4.5 フィードバック設計（視覚中心 + 触覚）✅
> 方針: **BGM・効果音は採用しない**。その分のリソースをビジュアル・演出・操作感に集中する。
> フィードバックは「視覚」を主、「触覚（振動）」を従とする。

| 事象 | 視覚 | 触覚(振動) |
| --- | --- | --- |
| PERFECT | 金閃光(控えめ)+収束リング着弾+方向フリック弧+二重衝撃波+火花大+スコアポップ+スローモー | 短2連 (10ms,10ms) |
| GOOD | 水色閃光+方向フリック+衝撃波+火花中+スコアポップ | 短1回 (15ms) |
| MISS | 朱フラッシュ+赤ビネット+シェイク+衝撃波 | 長1回 (40ms) |
| ボタン押下 | 押し込み+発光+タップ位置の波紋+矢印拡大 | なし |
| コンボ | プレイヤー周囲のオーラが連数で強まり色相シフト(水色→金) | なし |
| 攻撃接近 | 収束タイミングリング+脅威方向の矢印/エッジ発光+斬撃トレイル | なし |

- 触覚は `navigator.vibrate`（iOS Safari は未対応のため**フォールバック無振動許容**、
  設定でON/OFF）。視覚のみで完全に成立させる（触覚は加点要素）。
- **収束タイミングリング**: 攻撃のインパクト時刻 T にちょうどプレイヤー外周へ着弾する
  同心リング。「いつ弾くか」を直感的に提示し、操作の公平性・気持ちよさを担保する中核演出。

### 4.6 カラー＆アクセシビリティ ✅
- 色だけに依存しない: 方向は**矢印アイコン**、判定は**テキスト＋形**でも区別。
- コントラスト比 4.5:1 以上を本文テキストで確保。
- 設定で「振動ON/OFF」「モーション低減（スローモー/シェイク/閃光抑制）」。
- `prefers-reduced-motion` を尊重し、初期値に反映 🔁。

### 4.7 オンボーディング ✅
- 初回起動 or「遊び方」: 2〜3枚のミニチュートリアル（GIF的アニメ）。
  「左から来たら→右」「ちょうどで弾く」を体験させる**1回の練習パリィ**を必須化。
- READY 中に最小ヒント（矢印と"反対を押す"）を1秒表示。

## 5. 難易度・進行設計（GD）

### 5.1 進行方式 ✅
- **エンドレス＋ウェーブ昇格制**。一定成功数ごとに難易度ティアが上がる。
- 同一ティア内でも攻撃間隔を乱数で揺らし、単調化を防ぐ。

### 5.2 難易度パラメータ（ティアごとに変化, 🔁）
| パラメータ | 易 → 難 | 説明 |
| --- | --- | --- |
| 可視時間（予兆＋接近の合計） | 1000ms → 450ms | 反応猶予。短いほど難 |
| 攻撃間隔 | 1100ms → 550ms | 次の攻撃までの間。 |
| 方向数 | 2（左右）→ 4（上下追加） | T4 以降で上下解禁。 |
| フェイント率 | 0% → 25% | 予兆方向を一度変える（⏳/上位ティア限定）。 |
| 連続攻撃 | 単発 → 2連 | 短間隔の連続。 |

> 判定窓 `PERFECT/GOOD_WINDOW` は基本固定（理不尽回避）。難化は「速度・密度・選択肢」で行う。

### 5.3 ティア定義（例, 🔁）
| ティア | 解禁条件(累計成功) | 可視時間 | 方向 | 備考 |
| --- | --- | --- | --- | --- |
| T1 | 0 | 1000ms | 左右 | チュートリアル相当 |
| T2 | 8 | 850ms | 左右 | 間隔短縮 |
| T3 | 20 | 700ms | 左右 | 2連登場 |
| T4 | 36 | 600ms | 上下左右 | 選択肢倍化 |
| T5 | 56 | 500ms | 上下左右 | フェイント低確率 |
| T6+ | 80+ | 450ms 床 | 上下左右 | 速度床・密度最大 |

## 6. 技術要件（FE）

### 6.1 技術スタック ✅
- **ビルドレス・依存ゼロ**の素の Web（HTML5 + CSS + Vanilla JS ES Modules）。
  - 理由: iOS Safari で即動作、GitHub Pages 等に静的配信可能、保守容易。
- 描画: **Canvas 2D**（ゲーム本体）＋ DOM/CSS（HUD・メニュー）のハイブリッド。
  - Canvas: 敵/攻撃/パーティクル/閃光など高頻度更新。
  - DOM: ボタン・テキスト・メニュー（アクセシビリティ＆レイアウト容易）。
- 音: **不採用**（BGM・効果音なし。リソースは視覚・演出・操作感へ集中）。
- 状態保存: `localStorage`（ベストスコア・設定）。

### 6.2 ループ / 時間管理 ✅
- `requestAnimationFrame` 駆動。判定は**経過時刻ベース（performance.now()）**で行い、
  フレームレート非依存にする（30/60/120fps いずれでも同一判定）。
- 入力時刻は `event.timeStamp`（無ければ `performance.now()`）で記録。

### 6.3 入力 ✅
- `pointerdown`（Pointer Events）を主とし、未対応環境は `touchstart` フォールバック。
  - **`click` は使わない**（300ms遅延・遅延感の回避）。
- `touch-action: none`、`-webkit-touch-callout: none`、`user-select: none`。
- ダブルタップズーム/ピンチズーム抑止（viewport meta ＋ `touchstart` preventDefault）。
- スクロール・バウンス抑止（`overscroll-behavior: none`、`position: fixed` body）。
- PC: 矢印キー/AD・←→キーでも操作可（任意, 🔁）。

### 6.4 表示・解像度対応 ✅
- `<meta name="viewport" content="width=device-width, initial-scale=1, maximum-scale=1, user-scalable=no, viewport-fit=cover">`
- Canvas は `devicePixelRatio` を考慮し論理サイズ×DPR で描画（DPR 上限3でメモリ抑制）。
- 100vh 問題対策: `window.innerHeight` / `visualViewport` 実測 or `100dvh` 併用。
- セーフエリア `env(safe-area-inset-*)` をレイアウトに反映。
- 画面回転（横）検知時はプレイを一時停止し縦持ち誘導 🔁。

### 6.5 オーディオ ✅
- **不採用**。BGM・効果音は持たない。iOS の自動再生制限・無音判定の懸念をそもそも回避し、
  リソースを視覚・演出・操作感に集中する。

### 6.6 パフォーマンス目標 ✅
- iPhone SE2/3 で **安定 60fps**、SE1 で **45fps 以上**。
- パーティクルは**オブジェクトプール**で GC スパイク回避。フレーム内 new を最小化。
- 重い CSS（大きな blur/filter のアニメ）は常用しない。閃光は短時間・限定的。
- 初期ロード（初表示まで）: 3G 相当でも 3 秒以内（総アセット小）。

### 6.7 ディレクトリ構成（実装ガイド）✅
```
/index.html
/src/css/style.css
/src/js/
  main.js          # 起動・rAFループ・状態遷移
  config.js        # チューニング定数（窓/スコア/ティア）
  game.js          # ゲーム状態機械・進行
  enemy.js         # 攻撃の生成・ライフサイクル・タイミング
  input.js         # ポインタ/キー入力の正規化と時刻記録
  judge.js         # 3段階判定ロジック（純粋関数, テスト対象）
  scoring.js       # スコア/コンボ計算（純粋関数, テスト対象）
  renderer.js      # Canvas 描画
  particles.js     # パーティクルプール
  ui.js            # DOM HUD/メニュー更新
  storage.js       # localStorage ラッパ

/docs/
```

## 8. 受け入れ基準（QA）✅

機能・体験が満たすべき確定基準（v1リリース判定）:

1. iPhone SE2 実機/同等エミュレーション（375×667）でレイアウト崩れなく全画面操作可能。
2. iPhone SE1（320×568）でボタン/HUDが重ならず操作可能。
3. タップのみで TITLE→プレイ→ゲームオーバー→リトライの一周が完結する。
4. 判定が **PERFECT/GOOD/MISS の3段階**で、方向違いは MISS になる。
5. 早すぎ・遅すぎ・無入力がいずれも MISS として正しく扱われる。
6. 連打しても1攻撃につき1回しか判定されない。
7. スコア・コンボ・最大コンボ・PERFECT率がリザルトに正しく表示される。
8. ベストスコアが再訪時も保持される（localStorage）。

## 9. テスト観点（QA）

### 9.1 判定ロジック（純粋関数・自動テスト）
- Δ=0 → PERFECT、Δ=±PERFECT_WINDOW 境界、±GOOD_WINDOW 境界の内外。
- 方向違い × タイミング完璧 → MISS。
- 無入力（タイムアウト）→ MISS。

### 9.2 スコアロジック（純粋関数・自動テスト）
- コンボ倍率の段階（4,5,9,10連…）と上限 ×4.0。
- タイミングボーナスの上限/下限（0〜50）。
- MISS でコンボ0・加点0。

### 9.3 端末/ブラウザ手動テスト
- iPhone SE2/SE1（Safari）、iOS Chrome、Android Chrome、PC。
- 回転、バックグラウンド復帰（タブ離脱→復帰でポーズ）、低電力モード。

### 9.4 入力遅延キャリブレーション
- 既定窓で「理不尽でない」かのプレイテスト。必要なら端末ごとオフセット補正 🔁。

### 9.5 パフォーマンス
- 連続パーティクル発生時のFPS、長時間プレイ（5分）でのメモリ増加なし。
